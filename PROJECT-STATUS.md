### Walking-distance estimation
- Walking time from a stop's train station to a listing or event venue used to come from **Google Distance Matrix**. After unexpected Google API charges (see "Incidents") that integration was replaced with a free, in-process algorithm in `backend/walk.js`:
  - `haversineKm(lat1, lng1, lat2, lng2)` returns great-circle distance in km.
  - `walkMinsBetween(...)` multiplies by `DETOUR_FACTOR` (1.25, the typical small-town walking-path-to-straight-line ratio) and `MIN_PER_KM` (13, a comfortable 4.6 km/h pace) and floors at 1 minute. Output is within roughly 2 to 3 minutes of Google's old answers along the Northlander route.
  - `geocodeNominatim(address)` resolves a postal address to lat/lng via OpenStreetMap's free Nominatim service. Required for the events sync because submitted events only carry an address string, not coordinates.
- **`backend/sync-from-airtable.js`** (the listings cron): pure Haversine, no geocoder needed because every listing already has lat/lng from earlier enrichment runs. `getWalk` now calls `walkMinsBetween` directly. No external request per listing.
- **`backend/sync-events.js`** (the events cron): calls `geocodeNominatim(address)` then `walkMinsBetween`. The caller sleeps 1100 ms between attempts (success or miss) so the run stays under Nominatim's 1 request/second cap. The Nominatim helper sends a `User-Agent: NorthlanderGuide/1.0 (https://northlanderguide.com)` so OSM can identify the traffic and contact us if needed; per their policy, do not remove it.
- **Both cron workflows are re-enabled** on their original schedules (`sync-listings.yml` at 6:15 / 18:15 UTC, `sync-events.yml` at 7:15 / 19:15 UTC). They no longer hit any paid API.
- `GOOGLE_PLACES_KEY` is no longer read by either cron script. The env var stays in the workflow YAMLs for now so re-enabling Google later is a one-line change. The manual scripts (`enrich-listing.js`, `backfill-listings.js`, `update.js`, `build-static.js`) still call Google Places + Photos and are paused until the billing source is identified.

### 2026-06-01: Google API runaway charges
- **Symptoms**: $396 in Google Cloud charges accrued, projection trending toward $2000. User disabled all Google APIs in the Cloud console and deleted the offending credential.
- **Live website was not the source**: stop-page maps already used OpenStreetMap tiles via Leaflet, the "Get directions" links on listing cards built outbound `google.com/maps/dir/?...` deep links (consumer Google Maps, no API key, no cost), and `googleapis.com` references on every public page were Google Fonts CSS only. - **Likely cost source**: the two cron-scheduled workflows (`sync-events.yml` and `sync-listings.yml`) that called Distance Matrix on every listing/event without a cached `walkMins`. The `build-static.js` and `enrich-listing.js` scripts also call Places Photos (the most expensive endpoint at ~$7 / 1000 calls) but they only run on manual dispatch.
- **Remediation (committed)**: paused both cron triggers in commit `052dcd0`, then replaced the Google Distance Matrix integration with the `backend/walk.js` Haversine + Nominatim algorithm described above. Re-enabled the cron triggers afterward. The crons no longer hit any paid API.

## Northlander.app

The marketing page at `/plan` claims four App-Guide integrations and a ten-feature Carry-On / First Class split. We're building the actual app under `northlander-app/` as a free-tier-only PWA on top of Vercel + SvelteKit, deployed at `northlander.app` (domain owned at Namecheap, hosting still to be wired). No backend, no accounts, no payments for the MVP - everything lives on the user's device via IndexedDB. No tier paywall is implemented yet; every feature is available to every user.

### Data model (Dexie schema v4)
- **trips** (v1): `{ id, name, stopIds[], stops?, returnStops?, returnDate?, returnStopId?, color, strap, colorId, departureDate?, direction?, coverBlob?, createdAt, updatedAt }`. `id` is a slug from the trip name and is kept stable across rename so child rows don't orphan. **The authoritative route shape since 2026-06-03 is `stops: [{ stopId, date }]`** in chronological order (departure first). **Return leg became multi-stop on 2026-06-07**: `returnStops: [{ stopId, date }]` is an array of return entries chronologically after the outbound leg, and the legacy `returnDate` / `returnStopId` are now derived mirrors of the LAST return entry so cover ticket, recap, poster, and PDF print readers that haven't been taught about multi-stop returns continue to render a single Return chip without breaking. For back-compat, the trip store also mirrors `stopIds[]` (just the outbound ids), `departureDate` (= `stops[0].date`) and `direction` (derived: northbound if the farthest stop is north of the departing one) onto the row. Older trips that only have `stopIds[]` keep rendering; the trip page's `deriveStops` function falls back to those when `stops` is missing. `direction` is `'northbound' | 'southbound'` (defaults to northbound).
- **packingItems** (v1): `{ id (auto), tripId, name, packed, listName?, createdAt, updatedAt }`. **Optional unindexed `listName`** (added 2026-06-06) pins a row to a named packing list ("Mom" / "Kids" / "Camera bag"). Rows with `listName: null` belong to the trip's default unnamed list. The trip row carries the list of named lists in `extraPackingLists: string[]` and an optional display name for the default list in `defaultPackingListName: string`. All additive, no Dexie bump.
- **bookings** (v1): `{ id (auto), tripId, title, kind, status, dueDate?, createdAt, updatedAt }`. `kind` is `train | room | meal | activity | other`. `status` is `pending | booked` in the schema; **new rows default to `'booked'`** since 2026-06-06 because the App's UI dropped the Pending state entirely (it confused users for no payoff). The field stays in the schema so the print view's Booked stamp still has a value to read; legacy rows that were saved as `'pending'` under the old toggle keep their value. **Room bookings also carry optional unindexed fields** `checkIn`, `checkOut`, `address`, `contact`, `confirmation`, `notes` for the accommodation expand panel. They live as JSON properties on the row so no Dexie bump was needed; the `updateBooking(id, patch)` helper validates and trims them on the way in. Every booking also carries an **optional unindexed `startTime`** (HH:MM, 24h) added 2026-06-02 to drive chronological ordering; `normalizeStartTime` validates it on patch and `sortByStartTime` returns timed rows in HH:MM order followed by untimed rows in creation order. Bookings written by AddPlanModal also carry an unindexed `listingKey` (`stopId|listing-name-slug`) so the modal can flip the row to an "Added" checkmark on re-render.
- **diaryEntries** (v1, active): `{ id (auto), tripId, stopId?, text, entryDate?, createdAt, updatedAt }`. Body text + optional pin to a stop. `entryDate` (YYYY-MM-DD, added 2026-06-03) is the user-supplied "when this note happened" date; the composer defaults it to today but accepts backdated entries. When present it overrides `createdAt` for display and sort. Optional + unindexed, so legacy rows fall through to `createdAt`.
- **bucketItems** (v2): `{ id (auto), name, kind, stopId?, address?, url?, source?, notes?, createdAt, updatedAt }`. Global wishlist NOT scoped to a trip. Same five-kind palette as bookings so icons stay consistent.
- **budgetEntries** (v3): `{ id (auto), tripId, label, amount, category, stopId?, createdAt, updatedAt }`. Per-trip ledger. `category` is one of five (`transport | lodging | food | activities | other`) with its own brand color. **Optional unindexed `stopId`** (added 2026-06-03) pins a spend to a chapter so each per-stop BudgetTracker shows its own running tab; the cover stats `Spent` total still aggregates trip-wide. addBudgetEntry / updateBudgetEntry accept it; existing rows keep working unscoped.
- **photos** (v4): `{ id (auto), tripId, stopId?, blob, thumb, caption, width?, height?, takenAt?, createdAt, updatedAt }`. Originals resized to 1600 long-edge JPEG q=0.85 and a pre-baked thumbnail at 400 long-edge - both stored as Blobs on the row.
- **userEvents** (v5, added 2026-06-06): `{ id (auto), tripId, stopId?, name, startDate, startTime?, venue?, address?, url?, description?, createdAt, updatedAt }`. User-added events that sit above the Guide pull on each chapter's "Happening" Drawer. `startDate` is YYYY-MM-DD and indexed so the per-trip list query can sort cheaply; `startTime` is optional HH:MM 24h. All free-text fields trim + clamp to 500 chars on the description.
- `deleteTrip(id)` wraps the cascade cleanup (trips + packingItems + bookings + diaryEntries + budgetEntries + photos + userEvents) in one Dexie transaction. **Every new trip-scoped table MUST be added to the deleteTrip transaction or it will leak orphans.** `bucketItems` is intentionally NOT in the cascade because it's not trip-scoped.

### Hosting + deploy
- **Domain**: `northlander.app` is registered at Namecheap with the DNS pointed at Vercel.
- **Vercel project**: separate from the Guide's project, with **Root Directory set to `northlander-app/`** so Vercel runs `npm install` + `vite build` from inside the app directory and the Guide's `site/` directory is ignored. `@sveltejs/adapter-vercel` handles framework detection - no `vercel.json` needed inside `northlander-app/`. Auto-deploys from `main`.
- **All four cross-site CTAs from the Guide** (`/plan` page CTAs, stop-page Plan Your Trip card, listing-card Add to Trip pills, topbar Plan a Trip pill) link to `https://northlander.app` / `https://northlander.app/add` / `https://northlanderguide.com/plan`.
- **Build constraint - vite-plugin-svelte version**: `@sveltejs/vite-plugin-svelte` major versions track Svelte's. We're on Svelte 4, so the plugin must be **`^3.1.2`** (Svelte-4 line). Bumping the plugin to v4 would silently require Svelte 5 and break `npm install` with an ERESOLVE peer conflict on Vercel. The original scaffold shipped with `^4.0.0` and produced this failure on the first deploy attempt (commit `393df96` is the fix). If we ever upgrade to Svelte 5, bump both at once.
- **Svelte 4 `{@const}` constraint**: `{@const}` must be the **immediate child** of a logic block (`{#if}`, `{:else if}`, `{:else}`, `{#each}`, `{:then}`, `{:catch}`, `<svelte:fragment>`, or a `<Component>`). Putting it inside a plain element like `<div>` compiles locally in dev but **fails the Vite production build**. The fix is to either move the `{@const}` to be a direct child of the surrounding logic block, or compute the value as a reactive `$:` derivation in `<script>`. Caught on deploy as a `ValidationError` in `StopPickerModal.svelte` (resolved in the same fix as the deploy attempt).
- **Svelte 4 `{@const}` reactivity gotcha**: a `{@const}` inside a `{#each items as item}` only re-evaluates when the each block re-iterates (i.e. when `items` changes). It does NOT track reactive state read inside a helper function call. `{@const stopBookings = bookingsAt(stop.id)}` where `bookingsAt` reads the outer `bookings` array will lock to the value from the FIRST render and stay stale forever, even as `bookings` mutates. Caught 2026-06-06 on the trip page: the chapter Drawer count badges said "1 plan" while the BookingChecklist body filtered the live array correctly, so the user saw a count that disagreed with what was rendered below. Fix is to read the reactive variable inline in the consuming expression (e.g. `count={bookings.filter((b) => b.stopId === stop.id).length}`) so Svelte sees `bookings` as a direct template dependency. The `is-on` flag in `PackingPickerModal` follows the same pattern (`{@const already = existingItemIds.has(k)}`).
- **Svelte 4 markup comment style**: directives like `svelte-ignore a11y-autofocus` must be **HTML comments placed before the element**, not JS-style block comments inside the opening tag. Production builds fail with `ParseError: Expected >` on something like `/* svelte-ignore a11y-autofocus */ autofocus` mid-tag, even though dev tolerates it. Use `<!-- svelte-ignore a11y-autofocus -->` on the line above the element instead.
